## Задача

Ключевая задача — разработать отказоустойчивую инфраструктуру для сайта, включающую мониторинг, сбор логов и резервное копирование основных данных. Инфраструктура размещается в Yandex Cloud.

Перед началом работы над курсовым заданием изучена [Инструкция по экономии облачных ресурсов](https://github.com/netology-code/devops-materials/blob/master/cloudwork.MD).

## Инфраструктура

Для развёртки инфраструктуры использованы **Terraform** и **Ansible**.

### Сайт

- Созданы две ВМ (`web1`, `web2`) в разных зонах доступности (`ru-central1-a`, `ru-central1-b`).
- На каждой ВМ развернут сервер Nginx в Docker-контейнере. ОС и содержимое ВМ идентичны.
- Использован набор статичных файлов для сайта.
- Балансировка нагрузки реализована с помощью Nginx на Bastion (вместо Application Load Balancer) для соблюдения квоты на публичные IP-адреса.
- Тестирование сайта выполнено через `curl -v http://<публичный IP Bastion>:80`.

### Мониторинг

- Создана ВМ (`prometheus`) для Prometheus.
- На каждой веб-ВМ установлены **Node Exporter** и **Nginx Log Exporter**.
- Prometheus настроен на сбор метрик с этих экспортеров.
- Создана ВМ (`grafana`) для Grafana.
- Grafana настроена на взаимодействие с Prometheus. Настроены дашборды с отображением метрик: Utilization, Saturation, Errors для CPU, RAM, дисков, сети, а также `http_response_count_total` и `http_response_size_bytes`. Добавлены необходимые пороговые значения (tresholds).

> **Скриншот дашборда Grafana:**  
> ![Grafana Dashboard](вставить изображение)

### Логи

- Создана ВМ (`elasticsearch`) для Elasticsearch.
- На веб-серверах установлен Filebeat (в Docker-контейнере), настроена отправка access.log и error.log Nginx в Elasticsearch.
- Создана ВМ (`kibana`) для Kibana, сконфигурировано соединение с Elasticsearch.

> **Скриншот Kibana:**  
> ![Kibana](вставить изображение)

### Сеть

- Развернут один VPC (`diplom-project-network`).
- Сервера web, Prometheus, Elasticsearch помещены в приватные подсети.
- Сервера Grafana и Kibana размещены в приватных подсетях, а Bastion — в публичной для обеспечения единственной точки входа.
- Настроены Security Groups для соответствующих сервисов на входящий трафик только к нужным портам.
- Настроена ВМ с публичным адресом (`bastion`), в которой открыт только один порт — 22 (SSH). Все Security Groups настроены на разрешение входящего SSH из этой security group. Реализована концепция bastion host.

### Резервное копирование

- Вместо платных снапшотов Yandex Cloud реализован бесплатный бэкап через **rsync** на Bastion.
- Настроена ежедневная задача (cron) для копирования важных файлов (`/etc`, `/var/log`, `/opt`, `/home/ubuntu`) в каталог `/opt/backup`.
- Ограничено время жизни бэкапов до 7 дней (автоматическая очистка старых копий).

## Архитектура и принятые решения

Визуальная схема взаимодействия компонентов системы представлена в разделе [Визуальная архитектура проекта](https://github.com/MindMaze74/diplom-project/blob/main/README.md#визуальная-архитектура-проекта).

### Сетевая структура и состав ВМ

- Создан один VPC с двумя зонами доступности.
- В каждой зоне имеются публичная подсеть (для Bastion) и приватная подсеть (для всех остальных сервисов).
- Для доступа в интернет из приватных подсетей настроен NAT-шлюз.
- Security Groups настроены по принципу минимально необходимых прав.

| Имя ВМ | Зона доступности | Роль | Публичный IP | Прерываемая |
|--------|------------------|------|--------------|-------------|
| `bastion` | `ru-central1-a` | SSH-шлюз, Nginx-прокси (балансировщик) | **Да** (единственный) | Да |
| `web1` | `ru-central1-a` | Веб-сервер (nginx в Docker) | Нет | Да |
| `web2` | `ru-central1-b` | Веб-сервер (nginx в Docker) | Нет | Да |
| `prometheus` | `ru-central1-a` | Сбор метрик (Prometheus в Docker) | Нет | Да |
| `grafana` | `ru-central1-a` | Визуализация (Grafana в Docker) | Нет | Да |
| `elasticsearch` | `ru-central1-a` | Хранилище логов (Elasticsearch в Docker) | Нет | Да |
| `kibana` | `ru-central1-a` | Просмотр логов (Kibana в Docker) | Нет | Да |

> **Примечание:** Все ВМ используют **прерываемые** инстансы (`preemptible = true`), что позволяет экономить до 70% стоимости аренды. Это допустимо для курсовой работы, так как инфраструктура не является критичной к перезапускам.

### Сервисная модель

- **Веб-серверы**: Nginx в Docker-контейнерах, балансировка через Nginx на Bastion.
- **Мониторинг**: Node Exporter и Nginx Log Exporter на веб-серверах; Prometheus собирает метрики; Grafana визуализирует их.
- **Логирование**: Filebeat (в Docker) на веб-серверах отправляет логи Nginx в Elasticsearch; Kibana предоставляет интерфейс для просмотра логов.
- **Резервное копирование**: Бесплатный бэкап через rsync на Bastion с хранением 7 дней.

### Принятые компромиссы

В ходе проектирования и реализации инфраструктуры был принят ряд решений, которые отступают от «идеальной» архитектуры, но являются обоснованными в условиях ограничений облачного провайдера, бюджета и целей проекта. Ниже приведены ключевые компромиссы с подробным обоснованием.

#### 1. Ограничение на количество публичных IP-адресов

- **Проблема**: В Yandex Cloud действует квота на скорость создания публичных IP-адресов – не более 1 IP в минуту. При попытке создать несколько ресурсов с публичными IP одновременно (Bastion, Grafana, Kibana, Application Load Balancer) возникала ошибка `rate exceeded`, что делало развёртывание невозможным.  
- **Варианты решения**:  
  - Запросить увеличение квоты в поддержке (получен отказ, так как это внутреннее ограничение сервиса).  
  - Создавать IP-адреса последовательно с большими задержками (неработоспособно из‑за жёсткого лимита).  
  - Использовать статические IP, зарезервированные заранее (требует ручных действий и не решает проблему скорости создания).  
  - **Принятое решение**: Единственный публичный IP выделен только для Bastion. Все остальные сервисы размещены в приватных подсетях, доступ к ним организован через Nginx-прокси на Bastion.  
- **Результат**: Квота не превышается, развёртывание проходит без ошибок. Безопасность повышена, так как количество публичных точек входа сведено к минимуму. Доступ к сервисам сохранён через прокси.  
- **Компромисс**: Потеряна прямая публичная доступность Grafana и Kibana, доступ к ним осуществляется через Bastion. Это стандартная практика для внутренних сервисов, повышающая безопасность.

#### 2. Отказ от Yandex Application Load Balancer

- **Проблема**: По заданию требуется балансировка нагрузки между веб-серверами. Создание управляемого Application Load Balancer (ALB) потребовало бы дополнительного публичного IP, что нарушило бы квоту. Кроме того, ALB — платный сервис, что увеличивает стоимость.  
- **Варианты решения**:  
  - Использовать Network Load Balancer (также требует публичного IP и дополнительной настройки).  
  - Использовать сторонний балансировщик (например, HAProxy) на отдельной ВМ (увеличивает количество ВМ и затраты).  
  - **Принятое решение**: Балансировка реализована с помощью Nginx на Bastion. Bastion и так является единственной точкой входа, и добавление балансировки не требует дополнительных ресурсов.  
- **Результат**: Балансировка работает, не требует дополнительных затрат и публичных IP. Nginx на Bastion также выступает прокси для Grafana и Kibana, что делает его центральным элементом маршрутизации.  
- **Компромисс**: Балансировщик не является управляемым сервисом Yandex Cloud, поэтому его администрирование (мониторинг, обновление, обеспечение отказоустойчивости) полностью лежит на нас. Это даёт полный контроль, но требует дополнительных усилий по сопровождению.

#### 3. Размещение Grafana и Kibana в приватных подсетях

- **Проблема**: Согласно исходному заданию, Grafana и Kibana должны находиться в публичных подсетях для прямого доступа из интернета. Однако это потребовало бы выделения им отдельных публичных IP, что невозможно из-за квоты.  
- **Варианты решения**:  
  - Выделить публичные IP, но нарушить квоту (приводит к ошибкам).  
  - Использовать Bastion как прокси, оставив сервисы в приватных подсетях.  
  - **Принятое решение**: Сервисы размещены в приватных подсетях, доступ к их веб-интерфейсам осуществляется через Nginx-прокси на Bastion (порты 3000 и 5601).  
- **Результат**: Функциональность не пострадала, доступ к веб-страницам сохранён. Безопасность повышена, так как сервисы не видны из интернета напрямую.  
- **Компромисс**: Отступление от буквы задания, но с обоснованием (квота, безопасность). Такой подход (reverse proxy для внутренних сервисов) широко применяется в реальных проектах.

#### 4. Использование прерываемых (preemptible) ВМ

- **Проблема**: Бюджет проекта ограничен, необходимо минимизировать затраты на облачные ресурсы.  
- **Варианты решения**:  
  - Использовать обычные (непрерываемые) ВМ – стоимость выше, но гарантированная доступность.  
  - **Принятое решение**: Все ВМ созданы с параметром `preemptible = true`, что даёт экономию до 70% стоимости.  
- **Результат**: Значительная экономия бюджета.  
- **Компромисс**: Прерываемые ВМ могут быть остановлены в любой момент (максимальное время жизни — 24 часа). Инфраструктура не гарантирует непрерывную работу, но при остановке сервисы будут автоматически перезапущены политикой Docker (restart: always).

#### 5. Автоматизация SSH-доступа

- **Проблема**: Для доступа к приватным ВМ через Bastion требуется наличие SSH-ключа на Bastion и на всех целевых ВМ. Ручное копирование ключей неавтоматизировано и создаёт риск ошибок.  
- **Варианты решения**:  
  - Ручное копирование ключей (неудобно, не соответствует DevOps-практикам).  
  - **Принятое решение**: Terraform передаёт публичный ключ на все ВМ через параметр `ssh-keys` в метаданных. Дополнительно Ansible распространяет ключи внутри сети, чтобы Bastion мог подключаться к остальным ВМ.  
- **Результат**: Доступ полностью автоматизирован, исключены ручные операции.  
- **Компромисс**: Для работы Ansible требуется, чтобы на Bastion уже был установлен SSH-ключ (это решается на этапе Terraform). В целом подход стандартный и безопасный.

#### 6. Отказ от управляемых сервисов Yandex Cloud (Managed Databases, Managed Kubernetes)

- **Проблема**: Использование управляемых сервисов (например, Managed Service for PostgreSQL, Managed Kubernetes) упростило бы эксплуатацию, но значительно увеличило бы стоимость проекта (оплата за каждый управляемый ресурс) и усложнило бы настройку (необходимость изучения дополнительных интерфейсов).  
- **Варианты решения**:  
  - Использовать Managed Databases для Prometheus (адаптер PostgreSQL) – дорого и избыточно для целей проекта.  
  - Использовать Managed Kubernetes для оркестрации контейнеров – требует дополнительных навыков и затрат.  
  - **Принятое решение**: Все сервисы (Prometheus, Grafana, Elasticsearch, Kibana) развёрнуты в Docker-контейнерах на собственных ВМ. Управление осуществляется через Ansible.  
- **Результат**: Полный контроль над конфигурацией, минимальные затраты, простота отладки.  
- **Компромисс**:  
  - **Нет автоматического масштабирования** – при росте нагрузки потребуется вручную увеличивать ресурсы ВМ или добавлять новые.  
  - **Нет автоматического failover (отказоустойчивости)** – при падении ВМ сервис станет недоступен до ручного перезапуска (или автоматического перезапуска Docker-контейнера). В отличие от управляемых сервисов, здесь нет встроенных механизмов репликации и автоматического переключения на здоровую ноду.  
  - Инфраструктура не обеспечивает промышленный уровень доступности (99.9%). В реальных условиях для гарантированной надёжности использовались бы Managed Kubernetes и Managed Databases.

#### 7. Выбор Docker вместо нативной установки сервисов

- **Проблема**: Установка каждого сервиса (Nginx, Prometheus, Grafana и т.д.) напрямую на ВМ (через apt или из исходников) требует больше времени на настройку, управления зависимостями и может привести к конфликтам версий.  
- **Варианты решения**:  
  - Нативная установка через пакетные менеджеры – требует ручной настройки каждой ОС, сложнее в обновлении и воспроизведении.  
  - **Принятое решение**: Все сервисы запущены в Docker-контейнерах с использованием официальных образов. Это обеспечивает единообразие окружения, упрощает обновление и переносимость.  
- **Результат**: Быстрое и воспроизводимое развёртывание, упрощённое управление версиями.  
- **Компромисс**: Docker добавляет небольшой оверхед по производительности и требует навыков работы с контейнерами. Однако в современных проектах это стандартный подход, который даёт больше преимуществ, чем недостатков.

## Развёртывание

### Предварительные требования
Перед началом убедитесь, что у вас есть:

Активный платёжный аккаунт в Yandex Cloud с достаточным балансом.

Установленные инструменты:

Terraform (>= 1.0) — [инструкция по установке](https://developer.hashicorp.com/terraform/tutorials/aws-get-started/install-cli)
Ansible (>= 2.9) — [инструкция по установке](https://docs.ansible.com/projects/ansible/latest/installation_guide/intro_installation.html)
Yandex Cloud CLI — [инструкция по установке](https://yandex.cloud/ru/docs/cli/operations/install-cli?utm_referrer=about%3Ablank)
Настроенный профиль Yandex Cloud CLI — выполните yc init и авторизуйтесь.
Права доступа - ваш аккаунт должен иметь роль editor или выше в каталоге, где будут создаваться ресурсы.
Сервисный аккаунт с авторизованным ключом (путь к ключу указывается в terraform.tfvars).
SSH-ключи на локальной машине (~/.ssh/diplom и ~/.ssh/diplom.pub).

### Шаги по развёртыванию

1.  **Клонировать репозиторий**:
    ```bash
    git clone https://github.com/MindMaze74/diplom-project.git
    cd diplom-project
    ```

2.  **Настроить переменные Terraform**:
    ```bash
    cd terraform
    cp terraform.tfvars.example terraform.tfvars
    ```
   >Отредактируйте файл terraform.tfvars, указав свои данные (ID облака, каталога, пути к ключам).

3.  **Проверить версии инструментов (рекомендуется)**:
    ```bash
    terraform --version
    ansible --version
    yc --version
    ```
4.  **Развернуть инфраструктуру**:
    ```bash
    terraform init
    terraform apply -parallelism=1
    ```
   > Дождитесь завершения (около 10–15 минут). При появлении запроса подтвердите действие вводом yes
   > Примечание: Параметр -parallelism=1 ограничивает параллельность для обхода квоты на создание публичных IP-адресов.

5.  **Проверить созданные ресурсы**:
    ```bash
    terraform state list
    terraform output
    ```
6.  **Настроить сервисы через Ansible**:
     ```bash
    cd ../ansible
    ansible-playbook -i inventory/inventory.ini playbooks/site.yml
    ```
    >Это займёт ещё 5–10 минут.

7.  **Проверить логи Ansible (если возникли ошибки)**:
    ```bash
    ansible-playbook -i inventory/inventory.ini playbooks/site.yml -vvv
    ```
8.  **Проверить доступность сервисов (см. раздел Доступ к сервисам).**


#### Доступ к сервисам

| Сервис | URL | Логин/Пароль |
|--------|-----|--------------|
| **Сайт** | `http://<bastion_public_ip>/` | – |
| **Grafana** | `http://<bastion_public_ip>:3000` | `admin` / `admin` |
| **Kibana** | `http://<bastion_public_ip>:5601` | – (без аутентификации) |
>Важно:

> ```
>Kibana может быть недоступна в течение первых 2–3 минут после запуска, так как ей требуется время для инициализации.

В результате работы спроектирована и реализована полностью автоматизированная инфраструктура, отвечающая минимальным требованиям курсового задания. Использование Terraform и Ansible обеспечивает воспроизводимость, а принятые компромиссы (единственный публичный IP, прокси на Bastion, прерываемые ВМ, отказ от управляемых сервисов) являются разумными и обоснованными в условиях облачных квот и ограниченного бюджета.

Инфраструктура готова к масштабированию: добавление новых веб-серверов или увеличение ресурсов ВМ не требует изменения архитектуры. Проект может служить основой для более сложных решений с использованием Managed Kubernetes, Managed Databases и других сервисов Yandex Cloud.
